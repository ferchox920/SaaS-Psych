# S1_1.1 - Auth Bearer Middleware + RBAC Guard

## Paso
`S1_1.1`

## Objetivo
Agregar middleware de autenticacion bearer (JWT access token), propagar `user_id/role` al contexto y aplicar guard RBAC reutilizable para proteger endpoints.

## Estado previo detectado
- El ultimo paso completado era `S1_1`.
- Ya existian endpoints `/api/v1/auth/login|refresh|logout`.
- Ya existia middleware de tenant (`X-Tenant-ID`) y contexto de tenant.
- No existia middleware para validar `Authorization: Bearer ...`.
- No existia guard de roles para autorizacion por RBAC.

## Cambios realizados
- Se agrego middleware `RequireAuth` que:
  - valida header `Authorization: Bearer <token>`
  - valida firma JWT del access token
  - valida que `tenant_id` del token coincida con el tenant del request
  - inyecta `Principal{user_id, role}` al contexto
- Se agrego middleware `RequireRole(roles...)` para control RBAC por rol.
- Se agrego acceso a `Principal` en contexto (`WithPrincipal`, `PrincipalFromContext`).
- Se extendio `AuthHandler` con endpoint `GET /api/v1/auth/me`.
- Se registraron rutas protegidas:
  - `GET /api/v1/auth/me` (requiere auth bearer)
  - `GET /api/v1/auth/admin-check` (requiere auth bearer + rol `owner|admin`)
- Se incorporaron tests unitarios para auth middleware y RBAC guard.

## Decisiones y trade-offs
- Se reutilizo `AccessClaims` actual para evitar duplicacion de esquema de token.
- La verificacion de firma usa `JWT_ACCESS_SECRET`; si cambia el secreto, tokens previos quedan invalidos.
- Se protegio `/auth/me` para validar end-to-end el middleware sin esperar endpoints de negocio.
- La autorizacion fina por recurso se deja para usecases futuros (defensa en profundidad).
