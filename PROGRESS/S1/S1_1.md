# S1_1 - Tenant Context + Auth Base

## Paso
`S1_1`

## Objetivo
Implementar la base de Sprint 1 con autenticacion multi-tenant (login, refresh con rotacion y logout) y middleware obligatorio de `X-Tenant-ID`, respetando separacion por capas (handler/usecase/infra).

## Estado previo detectado
- El ultimo paso completado era `S0_1`.
- Ya existian estructura base en `apps/api`, `GET /health`, config y conexion Postgres.
- Ya existian tablas `tenants`, `users`, `user_roles`, `refresh_tokens`.
- No existian endpoints de auth ni middleware de tenant.
- No existia logica de negocio de auth en capa `usecase`.

## Cambios realizados
- Se agregaron errores de dominio tipados en `internal/domain/errors`.
- Se implemento middleware `RequireTenant` para:
  - exigir header `X-Tenant-ID`
  - validar UUID
  - verificar existencia de tenant en DB
  - inyectar `tenant_id` en el `context.Context` del request
- Se implemento `usecase/auth` con:
  - `Login` (email/password + access JWT + refresh token)
  - `Refresh` (rotacion de refresh token)
  - `Logout` (revocacion de refresh token)
- Se implemento `TokenService` para access JWT + refresh token opaco hasheado (SHA-256).
- Se implementaron repositorios Postgres de tenant y auth con filtro por `tenant_id` en todas las queries.
- Se agregaron handlers HTTP en `/api/v1/auth/login|refresh|logout` sin logica de negocio (delegan en usecase).
- Se actualizo wiring en `cmd/server/main.go` para registrar middleware y auth cuando hay DB.
- Se extendio config con secretos JWT y TTLs.
- Se agregaron tests unitarios para middleware tenant y token service.

## Decisiones y trade-offs
- Se uso refresh token opaco hasheado en DB (no JWT) para facilitar revocacion puntual y reducir exposicion.
- Se mantuvo JWT solo para access token.
- Se habilitan endpoints de auth solo cuando hay DB configurada, para mantener boot local sin caida dura.
- RBAC completo y middleware de auth bearer se difieren para siguiente subpaso de Sprint 1.
