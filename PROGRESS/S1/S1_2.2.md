# S1_2.2 - Auth Flow Integration Test contra Postgres real

## Paso
`S1_2.2`

## Objetivo
Agregar un test de integracion del flujo auth usando repositorios reales de Postgres (no in-memory), reutilizando migraciones y seeds del proyecto.

## Estado previo detectado
- El ultimo paso completado era `S1_2.1`.
- Ya existia test de integracion HTTP auth con repo en memoria.
- Base docker + migraciones + seeds ya habian sido aplicadas previamente.
- Aun no habia prueba de auth flow conectada a Postgres real.

## Cambios realizados
- Se agrego `TestAuthFlowPostgresIntegration` en `internal/http`:
  - usa `db.NewPostgresPool`
  - usa `TenantRepository` + `AuthRepository` reales
  - ejecuta `login -> me -> refresh -> logout -> refresh(401)` sobre seeds reales
- El test se ejecuta solo si `RUN_PG_INTEGRATION=1` para no romper `go test` local/CI sin DB.
- Se definio DSN por defecto con `127.0.0.1` para evitar resolucion ambigua de `localhost`.

## Archivos tocados
- `apps/api/internal/http/auth_flow_postgres_integration_test.go`
- `PROGRESS/PROGRESS_INDEX.md`

## Migraciones
- No aplica en este paso (se reutilizan `000001` + `000002` ya existentes).

## Comandos para ejecutar
```bash
cd apps/api
go test ./...
go build ./...
```

```bash
# Ejecucion opcional del test contra Postgres real
$env:RUN_PG_INTEGRATION='1'
$env:DATABASE_URL='postgres://sessionflow:sessionflow@127.0.0.1:5432/sessionflow?sslmode=disable'
go test ./internal/http -run TestAuthFlowPostgresIntegration -v
```

## Decisiones y trade-offs
- Se uso flag por env para mantener estabilidad del pipeline sin dependencia obligatoria de DB.
- Se eligio prueba E2E de auth en capa HTTP para validar wiring completo (middleware + usecase + repos).
- Se prefirio `127.0.0.1` en lugar de `localhost` para evitar resolucion a IPv6 con instancias inesperadas.
