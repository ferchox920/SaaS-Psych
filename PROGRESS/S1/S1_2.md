# S1_2 - Seed Demo + Auth Flow Integration Test

## Paso
`S1_2`

## Objetivo
Agregar datos demo multi-tenant por migracion y cubrir con test de integracion HTTP el flujo auth completo: `login -> me -> refresh -> logout`.

## Estado previo detectado
- El ultimo paso completado era `S1_1.1`.
- Ya estaban disponibles `login/refresh/logout` y middleware bearer + RBAC.
- El esquema inicial ya incluia tablas `tenants`, `users`, `user_roles`, `refresh_tokens`.
- No existian seeds demo para ejecucion local.
- No habia test de flujo completo auth a nivel HTTP.

## Cambios realizados
- Se agrego migracion `000002_seed_demo` con datos demo para 2 tenants y 2 usuarios.
- Se incluyeron roles (`owner`, `member`) asociados por tenant.
- Se uso `crypt(..., gen_salt('bf'))` (pgcrypto) para guardar password hash sin secretos hardcodeados.
- Se agrego test de integracion HTTP de auth:
  - login con tenant header
  - acceso a `/api/v1/auth/me` con bearer token
  - refresh con rotacion de refresh token
  - logout
  - verificacion de refresh fallido tras logout
- El test usa repositorio en memoria para validar wiring/middlewares/handlers/usecase sin dependencia de Postgres en CI local rapido.

## Decisiones y trade-offs
- Se eligio test de integracion HTTP con repo en memoria para mantener feedback rapido y estable.
- Se uso `crypt` de Postgres en migracion para no versionar hashes estaticos sensibles.
- Contrato actual de respuesta de auth conserva `PascalCase` (por serializacion default de Go struct); se mantiene para no romper pasos previos y se puede normalizar en siguiente iteracion.
