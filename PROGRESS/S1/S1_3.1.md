# S1_3.1 - Endpoint protegido GET /api/v1/audit

## Paso
`S1_3.1`

## Objetivo
Exponer endpoint protegido para consulta de auditoria con filtros y paginacion basica, respetando tenant isolation y RBAC.

## Estado previo detectado
- El ultimo paso completado era `S1_3`.
- Ya existia persistencia de eventos auth en `audit_logs`.
- No habia endpoint HTTP para leer auditoria.
- Ya existian middlewares tenant/auth/rbac listos para proteger rutas.

## Cambios realizados
- Se agrego `usecase/audit` con servicio de listado y validaciones de paginacion.
- Se extendio `AuditRepository` para listar registros desde `audit_logs` con filtros:
  - `tenant_id` (obligatorio)
  - `actor` (opcional)
  - `from` y `to` (opcionales RFC3339)
  - `limit` y `offset`
- Se agrego `AuditHandler` con endpoint:
  - `GET /api/v1/audit`
  - respuesta con `items` y `pagination`
- Se registro la ruta con proteccion:
  - `RequireTenant`
  - `RequireAuth`
  - `RequireRole(owner, admin)`
- Se agregaron tests minimos:
  - unit test de usecase (paginacion por default y validacion de limit)
  - unit test de handler (caso ok + actor invalido)
- Se conecto wiring en `main` para inyectar `AuditService` al servidor.

## Decisiones y trade-offs
- Se fijo `maxLimit=100` para limitar costo de consultas.
- Se mantuvo query SQL armada manualmente para avanzar rapido sin agregar sql builder.
- Se devolvio `count` por pagina (no total global) para simplificar MVP.
- Errores de validacion se responden como `400 validation_error`.
