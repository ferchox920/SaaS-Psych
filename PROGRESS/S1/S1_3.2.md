# S1_3.2 - Auditoria con total_count, order y cursor

## Paso
`S1_3.2`

## Objetivo
Mejorar `GET /api/v1/audit` agregando `total_count`, orden configurable y cursor basado en `created_at`, junto con test de integracion opt-in contra Postgres.

## Estado previo detectado
- El ultimo paso completado era `S1_3.1`.
- Endpoint de auditoria ya existia con filtros basicos y paginacion por `limit/offset`.
- Aun no habia `total_count` ni cursor para paginacion incremental.
- No habia test de integracion dedicado para endpoint de auditoria en Postgres real.

## Cambios realizados
- Se extendio `usecase/audit`:
  - `ListResult` con `TotalCount` y `NextCursor`
  - soporte `Order` (`asc|desc`) y `Cursor`
  - validacion de `order`
- Se extendio `AuditRepository`:
  - `ListAuditLogs` ahora soporta `order` y `cursor` en SQL
  - nuevo `CountAuditLogs` para `total_count`
- Se extendio `AuditHandler`:
  - query params nuevos: `order`, `cursor`
  - respuesta de paginacion ahora incluye `total_count` y `next_cursor`
- Se actualizaron tests unitarios de handler/usecase a la nueva firma.
- Se agrego test de integracion opt-in:
  - `TestAuditListPostgresIntegration` (`RUN_PG_INTEGRATION=1`)
  - valida login seed + consulta `GET /api/v1/audit?limit=5&order=desc`
  - valida `total_count` y payload de auditoria.

## Decisiones y trade-offs
- Cursor se basa en `created_at` para mantener implementacion simple en MVP.
- `total_count` se calcula con query separada para claridad (trade-off: una query extra).
- Se mantiene `offset` por compatibilidad con clientes ya existentes.
- Test de integracion real se deja opt-in para no bloquear entornos sin DB accesible.
