# S1_3 - Auditoria de acciones Auth en audit_log

## Paso
`S1_3`

## Objetivo
Registrar en `audit_log` las acciones de autenticacion (`login`, `refresh`, `logout`) manteniendo aislamiento por tenant y sin afectar la disponibilidad del flujo auth.

## Estado previo detectado
- El ultimo paso completado era `S1_2.2`.
- Ya existia auth flow completo con middleware tenant + bearer + RBAC.
- Ya existian tests de integracion in-memory y opt-in contra Postgres real.
- Aun no existia tabla `audit_logs` ni persistencia de eventos de auth.

## Cambios realizados
- Se agrego migracion `000003_audit_log` para crear tabla `audit_logs` multi-tenant.
- Se implemento `AuditRepository` (infra/db) para insertar eventos de auth en Postgres.
- Se agrego contrato `Auditor` en `usecase/auth` e inyeccion opcional en `Service`.
- Se registro auditoria de:
  - `auth.login.success`
  - `auth.refresh.success`
  - `auth.logout.success`
- Se mantuvo el flujo auth resiliente: fallas de auditoria no bloquean login/refresh/logout.
- Se actualizo wiring de `main` para inyectar `AuditRepository` al `AuthService`.
- Se agregaron tests unitarios en usecase para validar:
  - secuencia de eventos auditados en login->refresh->logout
  - tolerancia a error del auditor sin romper login.

## Migraciones
- `000003_audit_log.up.sql`
  - Crea tabla `audit_logs` con:
    - `tenant_id` (requerido)
    - `actor_user_id` (opcional, FK users)
    - `action`, `entity`, `metadata`, `created_at`
  - Crea indices por tenant/fecha y actor.
- `000003_audit_log.down.sql`
  - Elimina `audit_logs`.

## Decisiones y trade-offs
- Se uso `Auditor` opcional en usecase para mantener testabilidad y bajo acoplamiento.
- Se eligio no fallar auth si falla auditoria para priorizar disponibilidad.
- Se guarda `metadata` JSONB para evolucion de eventos sin romper esquema.
- `entity` se fija en `"auth"` en esta etapa para simplificar consultas iniciales.
