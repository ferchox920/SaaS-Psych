# S1_4.1 - action_prefix y paginacion multi-pagina en auditoria

## Paso
`S1_4.1`

## Objetivo
Extender `GET /api/v1/audit` con filtro por prefijo de accion (`action_prefix`) y validar paginacion real multi-pagina usando `next_cursor + next_cursor_id`.

## Estado previo detectado
- El ultimo paso completado era `S1_4`.
- Auditoria ya soportaba filtros `action/entity`, orden `asc|desc`, cursor por `created_at` con desempate por `id`.
- El endpoint ya devolvia `next_cursor` y `next_cursor_id`.
- Faltaba filtro por rango/prefijo de accion para consultas por familia de eventos.
- Faltaba prueba de integracion de segunda pagina con cursor compuesto.

## Cambios realizados
- Se extendio `ListFilter` del usecase de auditoria con `ActionPrefix`.
- Se extendio el handler HTTP para aceptar query param `action_prefix`.
- Se extendio el repositorio Postgres para aplicar `action LIKE '<prefix>%'` en:
  - `ListAuditLogs`
  - `CountAuditLogs`
- Se agrego test unitario en usecase para verificar forwarding de `action_prefix`.
- Se reforzo el integration test opt-in de auditoria:
  - genera multiples eventos auth (`login`, `refresh`, `logout`)
  - pagina 1 con `limit=1&action_prefix=auth&entity=auth`
  - pagina 2 usando `cursor` + `cursor_id`
  - valida no solapamiento entre paginas.

## Decisiones y trade-offs
- Se uso `LIKE '<prefix>%'` por simplicidad y claridad en MVP.
- Se mantiene compatibilidad de `action` exacto y `action_prefix` en paralelo.
- No se agrego indice especifico por prefijo en este paso para mantener cambio pequeno; puede evaluarse si crece volumen de auditoria.
