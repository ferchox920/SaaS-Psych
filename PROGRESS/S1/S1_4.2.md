# S1_4.2 - Validaciones de filtros y orden cronologico estricto en auditoria

## Paso
`S1_4.2`

## Objetivo
Agregar validaciones de combinaciones de filtros incompatibles en `GET /api/v1/audit` y cubrir orden cronologico estricto para `asc` y `desc` en test de integracion.

## Estado previo detectado
- Ultimo paso completado: `S1_4.1`.
- El endpoint de auditoria ya soportaba `action`, `action_prefix`, `entity`, `cursor`, `cursor_id`, `order`, `limit`, `offset`.
- Ya habia paginacion multi-pagina por cursor compuesto.
- Faltaban validaciones explicitas para combinaciones incompatibles.
- Faltaban asserts de orden cronologico estricto (`created_at`, `id`) en tests de integracion.

## Cambios realizados
- Se agregaron validaciones en `usecase/audit`:
  - `action` y `action_prefix` no pueden coexistir.
  - `cursor_id` requiere `cursor`.
  - `offset` no puede combinarse con `cursor`.
  - `from` debe ser menor o igual a `to`.
- Se ajusto `AuditHandler` para mapear estas validaciones a `400 validation_error`.
- Se agregaron tests unitarios del usecase para cada combinacion invalida.
- Se agrego test unitario del handler para validar error de servicio mapeado a `400`.
- Se reforzo integration test de auditoria (opt-in) para verificar orden estricto:
  - consulta `order=desc` y valida `created_at desc, id desc`.
  - consulta `order=asc` y valida `created_at asc, id asc`.
