# S1_4 - Filtros action/entity y cursor estable para auditoria

## Paso
`S1_4`

## Objetivo
Extender `GET /api/v1/audit` con filtros por `action` y `entity`, y robustecer el orden de paginacion para cursor con orden estable secundario por `id`.

## Estado previo detectado
- El ultimo paso completado era `S1_3.2`.
- El endpoint de auditoria ya soportaba `actor/from/to`, `limit/offset`, `order`, `cursor`, `total_count`.
- El cursor estaba basado solo en `created_at`, con riesgo de empates cuando dos eventos comparten timestamp.
- No existian filtros explicitos por `action` y `entity`.

## Cambios realizados
- Se extendio el filtro de usecase de auditoria:
  - nuevos campos `Action`, `Entity`, `CursorID`.
  - `ListResult` ahora incluye `NextCursorID`.
- Se extendio el handler HTTP:
  - nuevos query params `action`, `entity`, `cursor_id`.
  - validacion de `cursor_id` como UUID.
  - respuesta de paginacion incluye `next_cursor_id`.
- Se extendio el repositorio de Postgres:
  - filtros SQL por `action` y `entity`.
  - cursor compuesto `(created_at, id)` para desempate estable.
  - orden estable `ORDER BY created_at <order>, id <order>`.
- Se actualizaron tests:
  - unit test handler para `cursor_id` invalido.
  - unit test usecase para `NextCursorID`.
  - integration test opt-in para filtrar por `action=auth.login.success&entity=auth`.

## Decisiones y trade-offs
- Se mantiene `offset` por compatibilidad con clientes existentes, aunque cursor es preferible para listas grandes.
- Para cursor compuesto se reutiliza `cursor` (tiempo) y se agrega `cursor_id` opcional para no romper consumidores actuales.
- Si llega `cursor` sin `cursor_id`, el fallback usa `uuid.Nil`; esto preserva backward compatibility pero la navegacion mas robusta requiere ambos.
