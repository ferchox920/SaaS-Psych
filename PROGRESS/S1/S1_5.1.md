# S1_5.1 - Auth con errores tipados y mapeo HTTP unificado

## Paso
`S1_5.1`

## Objetivo
Migrar el flujo de `auth` al patron de errores tipados (`validation`, `unauthorized`, `forbidden`) y consolidar su mapeo HTTP con `errors.Is`.

## Estado previo detectado
- Ultimo paso completado: `S1_5`.
- Auditoria ya estaba migrada a errores tipados de validacion.
- Auth seguia con validaciones mayormente en handler y sin tests unitarios dedicados de usecase.
- `AuthHandler` ya mapeaba `unauthorized`/`validation`, pero no contemplaba `forbidden`.
- Faltaba consistencia entre usecase y handler para validaciones de entrada en auth.

## Cambios realizados
- Se agregaron validaciones tipadas en `usecase/auth`:
  - `tenant_id` requerido en `Login`, `Refresh`, `Logout`.
  - `email/password` requeridos en `Login`.
  - `refresh_token` requerido en `Refresh` y `Logout`.
  - todas devuelven `domainerrors.NewValidation(...)`.
- Se actualizo `AuthHandler`:
  - soporte explicito de `ErrForbidden` -> `403 forbidden`.
  - para `ErrValidation` ahora devuelve mensaje real (`err.Error()`) en lugar de texto generico.
- Se agregaron tests unitarios nuevos:
  - `usecase/auth/service_test.go` para validaciones tipadas en `Login/Refresh/Logout`.
  - `http/handlers/auth_test.go` para mapeo de `ErrValidation` y `ErrForbidden`.

## Decisiones y trade-offs
- Se movio validacion basica a usecase para no depender exclusivamente de HTTP handlers.
- Se mantuvo validacion en handler tambien (defensa en profundidad) para respuestas tempranas.
- No se introdujo un sistema global de serializacion de errores en este paso para mantener alcance incremental.
