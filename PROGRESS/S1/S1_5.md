# S1_5 - Errores tipados en auditoria y mapeo HTTP consolidado

## Paso
`S1_5`

## Objetivo
Reemplazar validaciones basadas en strings por errores tipados en el flujo de auditoria y consolidar el mapeo HTTP usando `errors.Is`.

## Estado previo detectado
- Ultimo paso completado: `S1_4.2`.
- El usecase de auditoria validaba filtros, pero devolvia errores de texto.
- El handler de auditoria clasificaba errores con `strings.Contains(...)`.
- Ya existia `ErrValidation` en `internal/domain/errors`, sin tipo dedicado con mensaje.
- Faltaba estandarizar el contrato de errores de validacion entre usecase y handler.

## Cambios realizados
- Se agrego tipo de error de validacion en dominio:
  - `ValidationError` con `Message`
  - `Is(target)` para compatibilidad con `errors.Is(err, ErrValidation)`
  - constructor `NewValidation(message string) error`
- Se actualizo `usecase/audit` para devolver `domainerrors.NewValidation(...)` en todas las reglas de validacion.
- Se actualizo `AuditHandler` para mapear validaciones con:
  - `errors.Is(err, domainerrors.ErrValidation)` -> `400 validation_error`
  - sin chequeos por texto.
- Se actualizaron tests:
  - usecase: se valida que error de orden invalido cumple `errors.Is(..., ErrValidation)`.
  - handler: test de validacion del servicio ahora usa `domainerrors.NewValidation(...)`.

## Decisiones y trade-offs
- Se centralizo el tipo de error en `domain/errors` para reutilizarlo en otros modulos.
- Se mantuvo el mensaje explicito de validacion en `Error()` para no perder detalle en la respuesta HTTP.
- No se hizo refactor global de otros handlers en este paso para mantener alcance incremental.
