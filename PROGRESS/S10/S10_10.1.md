# S10_10.1 - Estandarizacion de payload de errores HTTP con `error.details`

## Paso
`S10_10.1`

## Objetivo
Estandarizar respuestas de error HTTP para soportar `error.details` opcional y centralizar su construccion en un helper compartido, migrando primero auth y endpoints de dominio.

## Estado previo detectado
- El payload de error era consistente en `code/message`, pero no soportaba `details`.
- La construccion de errores HTTP estaba distribuida (principalmente `writeAPIError` en handlers).
- OpenAPI no documentaba `error.details`.

## Cambios realizados
- Se creo helper central de respuestas de error:
  - `apps/api/internal/http/response/error.go`
  - contrato: `{ "error": { "code", "message", "details?" } }`
- Se creo wrapper en handlers para mantener migracion incremental:
  - `apps/api/internal/http/handlers/error_response.go`
  - `writeAPIError(...)` delega en helper central.
- Se migro prioridad auth + dominio (sin cambiar status codes):
  - `auth.go`, `client.go`, `appointment.go`, `session_note.go`
  - se agregaron `details` en validaciones representativas (ej: `field`, `fields`).
- OpenAPI actualizado:
  - `ErrorResponse` ahora incluye `details` opcional.
  - ejemplo de `ValidationError` actualizado con `details.field`.
- Tests minimos agregados (formato error en casos representativos):
  - `TestWriteAPIError_WithDetails`
  - `TestAuthLogin_MissingCredentials_IncludesDetails`
  - `TestAppointmentCreate_InvalidClientID_IncludesFieldDetails`
  - `TestHandleDomainError_StillCompatibleWithoutDetails`

## Decisiones y trade-offs
- Se mantuvo `writeAPIError` como wrapper para evitar refactor masivo y reducir riesgo.
- `details` se agrega solo en casos de validacion claros; errores heredados sin detalles siguen compatibles.
- Middleware no se migro en este paso para respetar alcance incremental solicitado.
