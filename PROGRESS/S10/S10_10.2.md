# S10_10.2 - Hardening multi-tenant DB para user_roles

## Paso
`S10_10.2`

## Objetivo
Completar hardening multi-tenant en DB para `user_roles`, asegurando que la relacion hacia `users` respete `tenant_id` y rechazando inserciones cruzadas entre tenants.

## Estado previo detectado
- `user_roles` tenia FK simple `user_id -> users(id)`.
- Esto permitia inconsistencia potencial: `user_roles.tenant_id` podia no coincidir con el tenant real del usuario referenciado.
- El modelo no tiene tabla `roles`; los roles son un dominio acotado por `CHECK (role IN (...))`.

## Cambios realizados
- Nueva migracion forward-only `000010_user_roles_tenant_user_fk_hardening.up.sql`:
  - asegura indice unico compuesto `users(tenant_id, id)` (idempotente con `IF NOT EXISTS`).
  - elimina FK simple `user_roles_user_id_fkey`.
  - agrega FK compuesta `user_roles(tenant_id, user_id) -> users(tenant_id, id)` con `ON DELETE CASCADE`.
- Down migration `000010_user_roles_tenant_user_fk_hardening.down.sql` como no-op (forward-only).
- Nuevo test Postgres integration opt-in:
  - `TestUserRolesRejectCrossTenantUserReferencePostgresIntegration`.
  - valida que insertar `user_roles` con tenant A y user de tenant B falla con FK (`23503`).
  - valida insercion same-tenant exitosa.

## Decisiones y trade-offs
- Se uso FK compuesta para enforcement fuerte en DB sin refactor de app.
- Se reutilizo `users(tenant_id,id)` via indice unico compuesto (ya podia existir por migraciones previas; se mantiene idempotente).
- Sobre "roles": no hay tabla relacional `roles`; el control equivalente sigue siendo `CHECK` del campo `role`, sin cambio de modelo.
