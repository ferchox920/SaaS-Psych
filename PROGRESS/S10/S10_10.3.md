# S10_10.3 - Evaluacion y hardening de `audit_logs.actor_user_id` por tenant

## Paso
`S10_10.3`

## Objetivo
Evaluar y decidir si `audit_logs.actor_user_id` debe usar FK compuesta por tenant, e implementarla sin romper historial.

## Estado previo detectado
- `audit_logs.actor_user_id` tenia FK simple a `users(id)` con `ON DELETE SET NULL`.
- Esto permitia inconsistencia potencial: `audit_logs.tenant_id` podia diferir del tenant real del `actor_user_id`.
- `actor_user_id` nullable es requerido para eventos sin actor o referencias historicas no disponibles.

## Decision
Se implementa FK compuesta por tenant:
- `audit_logs(tenant_id, actor_user_id) -> users(tenant_id, id)` con `ON DELETE SET NULL`.
- Se conserva `actor_user_id` nullable para no romper historial/auditoria parcial.

## Cambios realizados
- Nueva migracion forward-only `000011_audit_logs_tenant_actor_fk_hardening.up.sql`:
  - asegura indice unico `users(tenant_id, id)` (idempotente).
  - backfill seguro: referencias cross-tenant existentes en `audit_logs.actor_user_id` se setean a `NULL`.
  - reemplaza FK simple por FK compuesta `audit_logs_tenant_actor_user_fkey`.
  - agrega indice parcial `idx_audit_logs_tenant_actor_user_id` para consultas por actor dentro de tenant.
- Down migration no-op (`000011_audit_logs_tenant_actor_fk_hardening.down.sql`) por politica forward-only.
- Test de integracion Postgres:
  - `TestAuditLogsRejectCrossTenantActorReferencePostgresIntegration`.
  - valida rechazo cross-tenant (`23503`), caso same-tenant exitoso y caso actor `NULL` exitoso.

## Decisiones y trade-offs
- Trade-off principal: para preservar historial, los casos historicos inconsistentes se convierten en `actor_user_id = NULL` en lugar de eliminar eventos.
- Se mantiene `ON DELETE SET NULL` para no perder trazas de auditoria cuando usuarios se eliminan.
- No se cambiaron queries/repositorios ni contrato de API.
