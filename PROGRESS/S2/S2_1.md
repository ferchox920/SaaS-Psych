# S2_1 - Clients MVP end-to-end (multi-tenant)

## Paso
`S2_1`

## Objetivo
Implementar `P1` para `Clients` de punta a punta: modelo de dominio, caso de uso CRUD, persistencia Postgres, endpoints REST protegidos y test de aislamiento por tenant.

## Estado previo detectado (rehidratacion)
- Ultimo paso completado en indice: `S1_5.2`.
- Auth, tenant middleware, RBAC y audit estaban implementados y con tests.
- No existian tablas ni endpoints de dominio `clients/appointments/notes`.
- `docker-compose` ya incluia `postgres` y `redis`, pero sin rate limit Redis activo en runtime.
- Observabilidad estaba parcial (solo `Recover` + logger basico de arranque, sin middleware de request logging ni `/metrics`).
- No se detecto pipeline CI en `.github/workflows`.

## Cambios realizados
- Se creo migracion de base para `clients` con `tenant_id` obligatorio e indices por tenant.
- Se agrego entidad de dominio `client` con validaciones (`tenant_id` y `fullname` obligatorios).
- Se implemento `usecase/client` con `Create/List/Get/Update/Delete` y validaciones de entrada.
- Se implemento repositorio Postgres `ClientRepository` con filtro por `tenant_id` en todas las queries.
- Se agrego `ClientHandler` HTTP con CRUD REST en `/api/v1/clients`.
- Se protegieron endpoints de clients con `TenantMiddleware + AuthMiddleware + RequireRole(owner|admin|member)`.
- Se agregaron tests:
  - unit de dominio (`internal/domain/client/client_test.go`)
  - unit de usecase (`internal/usecase/client/service_test.go`)
  - integration HTTP de tenant isolation (`internal/http/clients_integration_test.go`)
