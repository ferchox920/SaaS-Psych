# S2_2 - Appointments MVP end-to-end (multi-tenant + no-solapamiento)

## Paso
`S2_2`

## Objetivo
Implementar `P1` para `Appointments`: tabla y CRUD operativo (create/list by range/update/cancel), regla de no-solapamiento por tenant, y tests de aislamiento/rango.

## Estado previo detectado
- Ultimo paso completado: `S2_1` (Clients MVP).
- Auth + tenant middleware + RBAC + audit ya estaban disponibles.
- No existia tabla `appointments` ni endpoints `/api/v1/appointments`.
- No existia regla de negocio para evitar citas solapadas.

## Cambios realizados
- Se agrego migracion `appointments` con `tenant_id`, `client_id`, rango horario, `status`, `location` y `CHECK (starts_at < ends_at)`.
- Se agrego dominio `appointment` con:
  - validacion de rango horario
  - estados base (`scheduled`, `canceled`)
  - helper `Overlaps(...)`.
- Se implemento `usecase/appointment` con:
  - `CreateAppointment` (valida y bloquea solapamientos)
  - `ListAppointments(by range)`
  - `UpdateAppointment` (incluye check de solapamiento excluyendo el propio id)
  - `CancelAppointment`.
- Se implemento repositorio Postgres `AppointmentRepository` con filtro por `tenant_id` en todas las queries.
- Se agrego handler HTTP `AppointmentHandler` y rutas REST:
  - `POST /api/v1/appointments`
  - `GET /api/v1/appointments?from=&to=`
  - `PUT /api/v1/appointments/:id`
  - `POST /api/v1/appointments/:id/cancel`
- Se cableo en `server` y `main` con guard RBAC `owner|admin|member`.
- Se agregaron tests:
  - unit de dominio para solapamiento/rango
  - unit de usecase para conflicto por solapamiento
  - integration HTTP para tenant isolation + filtro por rango + conflicto `409`.

## Decisiones y trade-offs
- El no-solapamiento se aplico por tenant completo (tal como prioridad MVP), sin segmentar por profesional.
- Para mantener la iteracion pequena, no se agrego todavia auditoria de acciones de appointments; queda para el bloque posterior cuando se integre P2/P3.
