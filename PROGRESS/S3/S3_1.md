# S3_1 - Rate limiting Redis en /auth/login

## Paso
`S3_1`

## Objetivo
Implementar `P3`: rate limiting sobre `POST /api/v1/auth/login` usando Redis, con limite por IP y configuracion por variables de entorno.

## Estado previo detectado
- Dominio MVP (Clients + Appointments) y Session Notes ya estaban implementados (`S2_1` a `S2_3`).
- `docker-compose.yml` ya exponia Redis, pero no existia uso de Redis en runtime.
- Login de auth no tenia throttle contra fuerza bruta.

## Cambios realizados
- Se extendio `config` con:
  - `REDIS_URL`
  - `RATE_LIMIT_LOGIN_PER_MIN`
- Se agrego cliente Redis en `internal/infra/redis`:
  - parseo de URL
  - `Ping` de verificacion
  - store `LoginRateLimitStore` con `INCR` + `EXPIRENX` en ventana fija.
- Se agrego middleware HTTP `RequireLoginRateLimit`:
  - key por IP: `ratelimit:auth_login:ip:<real_ip>`
  - responde `429 too_many_requests` al exceder
  - responde `503 service_unavailable` si Redis falla
  - incluye headers `X-RateLimit-Limit` y `X-RateLimit-Remaining`.
- Se cableo en server:
  - `ServerDeps.AuthLoginRateLimit`
  - aplicado solo en `POST /api/v1/auth/login`.
- Se cableo en `main.go`:
  - inicializacion de Redis opcional cuando `REDIS_URL` y `RATE_LIMIT_LOGIN_PER_MIN > 0`
  - conexion y cierre graceful.
- Se agregaron tests unitarios del middleware para:
  - bloqueo al exceder limite
  - bypass cuando esta deshabilitado
  - error de store -> `503`.

## Decisiones y trade-offs
- Se uso ventana fija (INCR + EXPIRENX) por simplicidad MVP.
- Se mantuvo scope en login por IP, sin limitar otros endpoints todavia.
- Si Redis no se configura, el middleware no se activa (modo opt-in por entorno).
