# S3_2 - Hardening multi-tenant en appointments (client_id por tenant)

## Paso
`S3_2`

## Objetivo
Aplicar hardening multi-tenant en appointments para asegurar que `client_id` pertenezca al mismo `tenant_id`, con enforcement en DB y validacion defensiva en usecase.

## Estado previo detectado
- Sprint mas reciente: `S3`.
- En `S3` solo existia un reporte (`S3_1`), por lo que para rehidratacion de 2 reportes se tomo tambien `S2_3` como continuidad inmediata.
- Dominio MVP existia: clients, appointments y session notes.
- Auth + tenancy + RBAC existian y estaban operativos.
- Rate limit Redis en `/auth/login` existia (`S3_1`).
- Observabilidad (request logging, `/metrics`, tracing) seguia ausente/parcial.
- CI, README y SPRINTS.md seguian ausentes.
- Riesgo detectado: `appointments.client_id` referenciaba solo `clients(id)` sin amarre estricto por `tenant_id`.

## Cambios realizados
- Se agrego migracion de hardening:
  - indice unico compuesto en `clients (tenant_id, id)`.
  - reemplazo de FK simple por FK compuesta `appointments (tenant_id, client_id) -> clients (tenant_id, id)`.
- Se agrego validacion defensiva en usecase de appointments:
  - nuevo contrato de repo `ClientExists(tenantID, clientID)`.
  - `Create` y `Update` verifican pertenencia de client al tenant antes de persistir.
- Se implemento `ClientExists` en `AppointmentRepository` (query filtrada por tenant).
- Se ajustaron tests:
  - unit: create falla con `ErrNotFound` si client no pertenece al tenant.
  - integration: intento de crear appointment en Tenant A con `client_id` de Tenant B devuelve `404`.
- Se aclaro mensaje `not_found` en handler de appointments para cubrir client/appointment.

## Migraciones
- `000007_appointments_tenant_client_fk_hardening.up.sql`
  - `CREATE UNIQUE INDEX ... clients (tenant_id, id)`
  - `DROP CONSTRAINT appointments_client_id_fkey`
  - `ADD CONSTRAINT appointments_tenant_client_fkey FOREIGN KEY (tenant_id, client_id) REFERENCES clients (tenant_id, id)`
- `000007_appointments_tenant_client_fk_hardening.down.sql`
  - revierte a FK simple por `client_id`
  - elimina indice compuesto agregado

## Decisiones y trade-offs
- Se aplico doble defensa (DB + usecase) para cubrir consistencia y respuesta temprana.
- Se mantuvo codigo y contratos HTTP estables (sin cambios de endpoint).
- Se devolvio `not_found` para no revelar existencia de recursos en otro tenant.
