# S4_4.4 - Middleware de request logging con request_id en Echo

## Paso
`S4_4.4`

## Objetivo
Implementar middleware de logging por request en Echo con `request_id` (propagado desde header o generado), logging JSON con campos operativos clave y validacion minima de presencia de `request_id`.

## Estado previo detectado
- Arquitectura por capas activa (`domain/usecase/infra/http`) y wiring central en `main`.
- Dominio MVP implementado: clients, appointments y session notes.
- Auth + tenancy + RBAC + rate limit Redis ya implementados.
- Observabilidad base ya tenia middleware de request logging (`S4_1`), pero faltaba reforzar validacion automatica de `request_id` en test.
- No se detectaron logs de tokens/secrets en el middleware revisado.

## Cambios realizados
- Se reforzo test del middleware para validar explicitamente que el log JSON contiene `request_id`.
- Se mantuvo comportamiento del middleware:
  - respeta `X-Request-ID` entrante;
  - genera UUID si no llega header;
  - escribe `X-Request-ID` en response;
  - loguea `request_id`, `tenant_id` (si existe), `user_id` (si existe), `method`, `path`, `status`, `latency_ms`.
