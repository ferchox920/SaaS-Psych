# S5_5 - Auditoría de acciones de dominio desde usecases

## Paso
`S5_5`

## Objetivo
Implementar auditoría para acciones de dominio desde usecases (no handlers):
- `client.create/update/delete`
- `appointment.create/update/cancel`
- `session_note.create` (no existe `update` en el modelo actual)

## Estado previo detectado
- Auditoría existente para auth (`login/refresh/logout`) en `AuditRepository.RecordAuthEvent`.
- Usecases de dominio no emitían eventos de auditoría.
- Estructura `audit_logs` ya disponible y suficiente (sin agregar campos).

## Cambios realizados
- Se extendió `AuditRepository` con `RecordDomainEvent(...)` reutilizando la tabla y shape actual de `audit_logs`.
- Se integró auditoría en usecases:
  - `client`: create/update/delete
  - `appointment`: create/update/cancel
  - `sessionnote`: create
- Se mantuvo emisión no bloqueante (si auditor falla, no rompe operación de negocio), igual criterio que auth.
- Se pasaron `actor_user_id` desde handlers (via principal JWT) a los usecases de `client` y `appointment`.
- Se actualizó wiring en `main.go` para inyectar `auditRepo` en servicios de dominio.
- Se ajustaron tests/integration por cambios de firma (`NewService(..., auditor)` y métodos con actor).
- Se agregaron tests mínimos de usecase para validar que se emite audit event en acciones de dominio.
