# S7_7.2 - Instrumentacion OpenTelemetry de Postgres en infra/db

## Paso
`S7_7.2`

## Objetivo
Instrumentar Postgres con OpenTelemetry en `infra/db` para generar spans en operaciones DB (Query/QueryRow/Exec), con atributos minimos y control de `db.statement` por entorno.

## Estado previo detectado
- Tracing HTTP ya estaba implementado en `S7_7.1`.
- No habia spans de DB; `pgxpool` se creaba sin tracer.
- `config` no incluia flag para habilitar/deshabilitar `db.statement`.
- No habia test de integracion DB que validara emision de spans.

## Cambios realizados
- Se agrego instrumentacion de `pgx` via `QueryTracer` en `infra/db`:
  - archivo nuevo `internal/infra/db/tracing.go`
  - spans `postgres.query` por `Query/QueryRow/Exec`
  - atributos:
    - `db.system=postgresql`
    - `db.name` (parseado de `DATABASE_URL`)
    - `db.statement` solo si esta habilitado
  - error handling: `RecordError` + status `Error`.
- Se extendio creacion de pool:
  - `NewPostgresPoolWithTracing(ctx, databaseURL, PoolTracingConfig)`
  - `NewPostgresPool` se mantiene como wrapper para compatibilidad.
- Se cableo en `main.go` la inyeccion del tracer de DB:
  - `tracerProvider.Tracer("sessionflow/db")`
  - flag `OTEL_DB_STATEMENT_ENABLED`.
- Se agrego config/env:
  - `OTEL_DB_STATEMENT_ENABLED` en `config` y `.env.example`.
- Tests agregados:
  - unit: parseo de `db.name` y sanitizacion de SQL (`tracing_test.go`)
  - integration opt-in: verifica que se emite al menos un span DB con `RUN_PG_INTEGRATION=1` (`tracing_postgres_integration_test.go`).
- Se documento en README el trade-off de `db.statement` (privacidad/performance).

## Decisiones y trade-offs
- Se eligio tracer de `pgx` a nivel pool para cobertura transversal de repos y cambio minimo en arquitectura.
- `db.statement` queda deshabilitado por defecto para reducir riesgo de exponer datos sensibles y overhead de serializacion.
- Cuando se habilita, se aplica sanitizacion basica (normalizacion de whitespace + truncado) pero no ofuscacion semantica completa.
