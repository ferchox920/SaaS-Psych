# S8_8 - Correlacion logs-traces (trace_id/span_id en request logging)

## Paso
`S8_8`

## Objetivo
Agregar correlacion entre logs y tracing OpenTelemetry en middleware de request logging, incluyendo `trace_id` y `span_id` del contexto activo.

## Estado previo detectado
- Middleware de request logging ya incluia `request_id`, `tenant_id`, `user_id`, `method`, `path`, `status`, `latency_ms`.
- Tracing OpenTelemetry HTTP estaba activo en middleware separado.
- Faltaba correlacion directa logs-traces (`trace_id`/`span_id`) en logs de request.

## Cambios realizados
- Se actualizo `RequestLogging` para anexar campos:
  - `trace_id`
  - `span_id`
- Los campos se leen del contexto activo (`trace.SpanFromContext`):
  - si hay span valido: valores no vacios
  - si no hay span: strings vacios (`""`) para mantener compatibilidad estructural.
- Se mantuvo intacto el resto del formato de logs (`request_id`, `tenant_id`, `user_id`, `status`, `latency`).
- No se tocaron metricas ni middleware de metrics.
- Se agregaron tests:
  - sin tracing activo: `trace_id` y `span_id` presentes y vacios
  - con tracing activo: ambos campos presentes y no vacios.

- Se agregan los campos de forma no disruptiva para consumidores actuales de logs JSON.
