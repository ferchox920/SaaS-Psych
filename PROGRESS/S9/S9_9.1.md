# S9_9.1 - Hardening DB multi-tenant con FKs compuestas en session_notes y refresh_tokens

## Paso
`S9_9.1`

## Objetivo
Reforzar integridad multi-tenant en base de datos para evitar referencias cruzadas entre tenants en:
- `session_notes -> appointments`
- `refresh_tokens -> users`

## Estado previo detectado
- Ya existia hardening compuesto para `appointments (tenant_id, client_id)`.
- `session_notes` referenciaba `appointments(id)` solo por `appointment_id`.
- `refresh_tokens` referenciaba `users(id)` solo por `user_id`.
- En ambos casos podia existir riesgo de inconsistencia cross-tenant si se inyectaban IDs de otro tenant fuera del flujo normal.

## Cambios realizados
- Nueva migracion forward-only `000009_tenant_composite_fk_hardening.up.sql`:
  - crea indice unico `users(tenant_id, id)` para habilitar FK compuesta.
  - crea indice unico `appointments(tenant_id, id)` para habilitar FK compuesta.
  - reemplaza FK de `refresh_tokens.user_id` por FK compuesta `refresh_tokens(tenant_id, user_id) -> users(tenant_id, id)`.
  - reemplaza FK de `session_notes.appointment_id` por FK compuesta `session_notes(tenant_id, appointment_id) -> appointments(tenant_id, id)`.
- Down migration `000009_tenant_composite_fk_hardening.down.sql` definida como no-op (forward-only).
- Test de integracion Postgres opt-in nuevo:
  - valida que DB rechaza `session_notes` con `tenant_id` A apuntando a `appointment_id` de tenant B.
  - valida que DB rechaza `refresh_tokens` con `tenant_id` A apuntando a `user_id` de tenant B.
  - valida caso positivo same-tenant para `session_notes`.

## Decisiones y trade-offs
- Se eligio enforcement en DB (FK compuesta) para garantizar integridad incluso ante bypass de capa de aplicacion.
- Se agregaron indices unicos compuestos minimos para cumplir requisitos de referencia FK sin refactor de claves primarias.
- Down migration se dejo no-op para respetar enfoque forward-only de hardening.
