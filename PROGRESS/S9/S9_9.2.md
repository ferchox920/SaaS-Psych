# S9_9.2 - Formalizacion de lifecycle de appointments (cancel reemplaza delete)

## Paso
`S9_9.2`

## Objetivo
Formalizar el modelo de dominio de appointments documentando que el endpoint `cancel` reemplaza `delete` para preservar historial clinico.

## Estado previo detectado
- El comportamiento ya existia en codigo: endpoint `POST /api/v1/appointments/{id}/cancel` y sin endpoint delete para appointments.
- Faltaba explicitar esta decision de diseno en guias y contrato API.
- El estado de domain usa `scheduled` y `canceled`.

## Cambios realizados
- `PROJECT_GUIDELINES.md`:
  - Alcance MVP actualizado: appointments con lifecycle state (sin delete).
  - Contratos HTTP: se agrego `POST /appointments/:id/cancel`.
  - Nota de diseno: cancelacion reemplaza delete para preservar historial clinico.
  - Sprint 2 ajustado para reflejar lifecycle state.
- `docs/openapi.yaml`:
  - `info.description` ahora explicita que `cancel` reemplaza `delete`.
  - `POST /api/v1/appointments/{id}/cancel` con descripcion de lifecycle `scheduled -> canceled`.
  - Schema `Appointment.status` con descripcion y enum `[scheduled, canceled]`.
- `apps/api/internal/usecase/appointment/service.go`:
  - Comentario de diseno en `Cancel` aclarando decision de no hard-delete y preservacion de historial.
