# S9_9.4 - Stack de metricas local con Prometheus + Grafana

## Paso
`S9_9.4`

## Objetivo
Agregar stack completo de metricas para desarrollo local usando Prometheus y Grafana, sin modificar middleware de metricas existente.

## Estado previo detectado
- API ya exponia `/metrics` con Prometheus client.
- `docker-compose.yml` no tenia servicios de Prometheus/Grafana.
- Existia stack de tracing (OTel Collector + Jaeger) en `deploy/observability`.
- No habia dashboard versionado para metricas HTTP del API.

## Cambios realizados
- `docker-compose.yml`:
  - Se agrego servicio `prometheus` (`localhost:9090`) con config montada desde `deploy/observability/prometheus.yml`.
  - Se agrego servicio `grafana` (`localhost:3000`) con provisioning y dashboard montados.
  - Se agregaron volúmenes `prometheus_data` y `grafana_data`.
- Prometheus:
  - Nuevo archivo `deploy/observability/prometheus.yml`.
  - Scrape de `host.docker.internal:8080/metrics` (job `sessionflow-api`).
- Grafana provisioning:
  - `deploy/observability/grafana/provisioning/datasources/datasource.yml` (datasource Prometheus).
  - `deploy/observability/grafana/provisioning/dashboards/dashboard.yml` (carga de dashboards desde filesystem).
- Dashboard:
  - Nuevo `deploy/observability/grafana-dashboard.json` con paneles base:
    - tasa de requests por `method/path/status`
    - latencia `p50/p95`
    - error rate `5xx`
    - volumen de requests por status (15m)
- README:
  - Se documento cómo levantar stack de metricas.
  - Se agregaron URLs de acceso y verificacion de targets en Prometheus.
  - Se listaron archivos de configuración y dashboard.

## Decisiones y trade-offs
- Se uso `host.docker.internal:8080` para scrapear API corriendo en host local.
- Se agrego provisioning automatico en Grafana para evitar import manual del dashboard en cada entorno.
- Dashboard inicial prioriza seńal operativa basica; paneles avanzados quedan para iteraciones futuras.
